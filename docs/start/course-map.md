# Eight-week course map

<p class="lesson-lead">The default route balances concepts, guided practice, debugging, and a capstone. Expand or compress it by changing practice time—not by removing the mental models.</p>

<div class="lesson-meta">

## Default route

| Week | Concepts | Studio work | Evidence of learning |
| --- | --- | --- | --- |
| 1 | Browser model, document anatomy, text, links | Semantic campus article | DOM inspection + explanation |
| 2 | Semantics, media, tables, forms | Event registration form | Keyboard and validation check |
| 3 | CSS rules, selectors, cascade, inheritance | Cascade crime scene | Computed-style trace |
| 4 | Box model, type, color, units, normal flow | Profile card | Overflow and zoom tests |
| 5 | Positioning, Flexbox, Grid | Navigation + course dashboard | Layout choice defense |
| 6 | Responsive design, states, motion | Responsive landing page | Multi-width and reduced-motion check |
| 7 | Accessibility, debugging, maintainable CSS | Repair + refactor clinic | Before/after evidence log |
| 8 | Modern CSS, performance, capstone | Campus hub + live change | Demo, code walk-through, rubric |

## Four-week sprint

For a short module, keep these non-negotiable:

1. semantic document structure, links, images, and forms;
2. cascade, inheritance, box model, and normal flow;
3. Flexbox, Grid, and responsive constraints;
4. keyboard/focus basics and a systematic debugging loop; and
5. one small project with a live change and explanation.

Treat motion, cascade layers, container queries, architecture patterns, and deeper performance work as extensions. Do not replace the cascade or normal-flow lessons with a framework tutorial; students will otherwise lack the model needed to debug framework output.

## Full-semester expansion

Add time for:

- design analysis and content hierarchy;
- more form states and error communication;
- responsive images and media formats;
- print and user-preference styles;
- browser compatibility and progressive enhancement;
- design tokens and component inventories;
- Git collaboration and review;
- a multi-page capstone with user testing; and
- repeated debugging clinics using unfamiliar code.

## Assessment balance

| Evidence | Weight | What it reveals |
| --- | ---: | --- |
| Prediction and concept checks | 10% | Mental model before tool assistance |
| Progressive labs | 30% | Implementation and verification habits |
| Debugging practicum | 20% | Evidence-based diagnosis under uncertainty |
| Capstone | 30% | Integrated quality and requirement ownership |
| Live change + explanation | 10% | Whether the student can modify and defend the work |

Large authentic tasks may use AI with disclosure. Short concept checks can be completed without AI when the goal is to measure the student's own model.

## Suggested weekly rhythm

1. **Predict:** a five-minute code-reading question.
2. **Model:** a concise explanation and live inspection.
3. **Practice:** one isolated skill task.
4. **Break:** compare solutions and name tradeoffs.
5. **Build:** add the concept to a real artifact.
6. **Verify:** test with explicit acceptance criteria.
7. **Explain:** complete an exit ticket or short walk-through.

## What to delay

Avoid front-loading:

- memorization of many rare elements or properties;
- a large build system;
- a CSS framework that hides layout decisions;
- pixel-perfect copying without content or accessibility criteria;
- long prompt-engineering lectures detached from code; or
- compatibility tables that will become stale.

Teach students how to consult current references instead of pretending a frozen list will remain current.
