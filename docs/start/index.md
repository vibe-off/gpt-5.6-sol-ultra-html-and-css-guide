# Learn the language the browser already speaks

<p class="lesson-lead">Frameworks change. Models change. HTML, CSS, the DOM, and the browser's rendering rules remain the layer every web interface must eventually satisfy.</p>

This course is not a tour of tags and properties. It is training in **implementation, inspection, debugging, and ownership**. You will create pages with plain HTML and CSS so every decision remains visible. You will also use AI, but only inside a workflow that makes its claims testable.

## What you will be able to do

By the end of the course, you should be able to:

- build a meaningful HTML document without relying on a framework;
- explain how the DOM, cascade, box model, normal flow, Flexbox, and Grid affect the result;
- translate a small visual brief into a responsive page;
- build usable navigation, media, data tables, and forms;
- meet a practical accessibility baseline for semantics, keyboard use, focus, labels, contrast, and motion;
- diagnose HTML and CSS problems from browser evidence instead of changing random values;
- organize CSS with low-specificity rules, reusable decisions, and clear component boundaries;
- review AI-generated code for correctness, accessibility, and unnecessary complexity; and
- modify and explain your work during a live requirement change.

## The PAVE loop

Every substantial exercise follows the same sequence:

1. **Predict** — state what you think the browser will do and why.
2. **Ask** — request a focused explanation, critique, or hypothesis from AI or a peer.
3. **Verify** — inspect the DOM, computed styles, keyboard behavior, and requirements.
4. **Explain** — record why the final result works and what evidence supports it.

The important move is from “the code looks plausible” to “the behavior is supported by evidence.”

## How lessons are organized

Most lessons contain:

- a concrete outcome;
- a mental model;
- a small worked example;
- a predict-before-running question;
- a guided practice task;
- an AI critique or debugging prompt;
- a browser verification checklist;
- common failure patterns; and
- an exit ticket that asks you to explain the idea in your own words.

Pages are marked implicitly by their purpose: foundation lessons teach concepts, quality lessons strengthen judgment, projects combine topics, and the reference desk supports work in progress.

## What this course does not require

You do not need a paid model, a cloud API, a CSS framework, or a powerful computer. Every AI activity can be completed with a peer, a teacher-supplied response, or the [debugging decision tree](/reference/debugging-tree). The work itself stays useful even when no model is available.

::: tip Start with a baseline
Before the first lesson, build a tiny page without AI. Keep it. At the end of the course, rebuild the same idea and compare the structure, resilience, and reasoning—not just the appearance.
:::
