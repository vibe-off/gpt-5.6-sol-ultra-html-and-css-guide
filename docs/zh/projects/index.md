# 用证据完成实践

<p class="lesson-lead">项目会把多个概念放进真实的需求变化中综合运用。每次构建都包含一项刻意约束、一份证据记录，以及一次用自己的话讲清楚的环节。</p>

<div class="lesson-meta">
  <span>项目</span>

## 项目循环

1. **阅读任务说明。** 分清内容、行为和质量要求。
2. **建立结构模型。** 画出区域、标题、交互和数据关系。
3. **编写验收标准。** 让“做得好”可以被观察和验证。
4. **构建最小的完整切片。** 先写语义化 HTML，再写 CSS。
5. **测试这个切片。** 检查键盘操作、不同宽度、缩放、长内容，并收集浏览器证据。
6. **邀请他人评审。** 同伴或 AI 都使用同一套标准提出反馈。
7. **应用现场变更。** 在不重写整个文件的前提下修改设计。
8. **解释决策。** 展示证据、权衡，以及一个被否决的方案。

## 提交内容

每个项目都应包含：

```text
project-name/
├─ index.html
├─ styles.css
├─ images/
└─ evidence.md
```

`evidence.md` 包含：

```markdown
# 证据

## 验收结果
- [ ] …

## 浏览器观察
- …

## 无障碍检查
- …

## AI 使用披露
- 建议：
- 验证：
- 决策：

## 现场变更
- 改了什么：
- 为什么这样实现：
```

在合适的情况下，证据文件也可以改为课堂表单或口头讲解，但其中要求的思考过程不能省略。

## 保证作品真实性的 AI 政策

当学习目标是理解需求、实现、评审、调试和建立代码所有权时，可以使用 AI。学生必须披露有实质影响的 AI 使用，并验证其结果。如果目标是检验学生独立形成的心智模型，简短的预测练习可以要求不使用 AI。

AI 不得伪造测试结果、用户反馈、引用来源或浏览器证据。

## 教师设计原则：制造意外，而不是设陷阱

现场变更应揭示当前结构是否易于调整，而不是用隐藏要求惩罚学生。合适的变更包括：

- 标题长度变为原来的两倍；
- 卡片增加一个状态和第二个操作；
- 导航增加一项；
- 表单增加一条条件式帮助说明；
- 通过令牌更换配色主题；或者
- 双栏区域必须在狭窄父容器内正常工作。

学生应能够追踪变更的影响，并通过局部修改完成实现。
