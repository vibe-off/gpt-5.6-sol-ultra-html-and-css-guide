# 评估判断力，而不是打字速度

<p class="lesson-lead">评分量表奖励学生能够测试、修改并解释的成果。AI 使用必须透明，但它本身既不能证明作品质量，也不能证明存在学术不端。</p>

## 结课项目评分量表

| 维度 | 分值 | 满分证据 |
| --- | ---: | --- |
| 语义结构与内容 | 15 | 地标、层级、元素选择、链接、媒体、数据和表单准确 |
| 无障碍与表单可用性 | 20 | 键盘完成、可见焦点、标签、名称/状态、对比度、缩放、动态效果 |
| 布局与响应式稳健性 | 20 | 流式/内在设计，有目的地使用 Flex/Grid，完成长内容和多宽度测试 |
| CSS 推理与可维护性 | 15 | 局部选择器、清晰的令牌/边界、可解释的层叠、最少重复 |
| 调试与验证 | 15 | 可复现证据、可检验假设、最小修复、回归检查 |
| 需求与边缘情况 | 10 | 流程完整、状态正确、完成现场变更、没有无法解释的故障 |
| AI 使用披露与代码所有权 | 5 | 披露有实质影响的使用、验证结论、解释接受/拒绝的建议 |

## 表现等级

### 优秀

学生能预判边缘情况，把决策与浏览器证据相连，优先选择原生而简单的方案，通过局部修改完成现场变更，并且无需依赖原始提示词就能解释权衡。

### 熟练

主要任务正确且无障碍，布局在已测试条件下稳健，CSS 易于理解，证据支持主要决策；学生在少量提示下能够解释和修改作品。

### 发展中

页面在默认视图下能够工作，但在语义、键盘操作、响应式情况、层叠推理或证据方面仍有缺口。学生能说明部分决策，但修改时仍依赖反复试错。

### 起步阶段

结果不完整或脆弱，重要行为依赖学生无法解释的生成代码，并且缺少浏览器证据或验收测试。

## 不可妥协的分数上限

出现下列情况时，教师可以限制最高得分：

- 无法完成主要流程；
- 无法通过键盘使用导航或表单；
- 提交的证据是伪造的；
- 学生无法解释自己提交的源代码；或者
- 处理私人数据的方式违反课程政策。

请在项目开始前公布这些上限规则。

## 现场变更评分（相关类别中共 10 分）

| 证据 | 分值 |
| --- | ---: |
| 将变更重新表述为可测试标准 | 2 |
| 找出受影响的结构和样式 | 2 |
| 通过局部修改完成可工作的实现 | 3 |
| 运行一项相关回归检查 | 2 |
| 解释一项权衡 | 1 |

## 调试实践评分量表

每个案例：

- 复现的准确度：**20%**；
- 浏览器证据：**30%**；
- 假设与实验：**20%**；
- 最小且有意图的补丁：**20%**；
- 回归检查：**10%**。

这样的权重可以避免学生在不理解原因的情况下，仅提交一次碰巧奏效的属性修改。

## 口头讲解问题

选择两到三个：

1. 展示一项能够改善原生行为的语义选择。
2. 沿层叠过程追踪一个最终计算值。
3. 解释影响布局的一项内在尺寸。
4. 演示键盘焦点如何走完主要流程。
5. 展示一项现代 CSS 特性的回退方案。
6. 复现一个缺陷，并解释帮助你定位它的证据。
7. 描述一条你拒绝采用的 AI 建议。
8. 修改一个令牌，并在保存前预测会影响哪些界面。

## 反馈语言

反馈应尽量关联证据和下一步行动：

- “标签可见，但它的 `for` 值与输入框 ID 不匹配；请点击标签验证是否能激活控件。”
- “Grid 在 1440px 和 375px 下都能工作；请测试卡片接近最小尺寸的 620–760px 区间。”
- “状态规则需要更高特异性，是因为基础选择器包含 ID；请降低基础选择器的特异性，而不是继续提高状态规则。”

不要只说“让它看起来更专业”，却不给出可观察的标准。
