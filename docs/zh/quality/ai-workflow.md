# 将 AI 放进工程闭环

<p class="lesson-lead">最好的提示词不一定最长。它应向模型提供正确证据、限制任务范围，并产出能够测试的答案。</p>

<div class="lesson-meta">
  <span>AI 素养</span>
  <span>75 分钟</span>
  <span>规划 → 验证 → 负责</span>
</div>

## 让请求匹配当前阶段

### 规划

请求需求清单、语义化内容大纲、边界用例或验收标准。在理解页面内容与行为之前，不要索取最终代码。

### 学习

请求一个思维模型、两个概念的对比，或一道预测练习。随后在小型示例中检查实际行为。

### 生成

只请求一个边界明确的组件或规则集，并说明 HTML 语义、CSS 约束、支持的浏览器基线和验收标准。确保生成的改动易于审查。

### 调试

提供复现步骤与浏览器证据。让模型列出按可能性排序的假设和能够区分它们的实验，而不是重写整个文件。

### 审查

分别审查需求、语义、无障碍、布局稳健性、层叠复杂度、性能和不必要代码。

## 高信噪比请求

```text
角色：作为调试伙伴，而不是代码生成器。

上下文：
这是一个位于 328px 容器中的纯 HTML/CSS 活动卡片。

预期行为：
当视口宽度缩小到 320px 时，图片和文字仍必须留在卡片内。

实际行为：
页面产生了 92px 的水平溢出。

浏览器证据：
图片的计算宽度是 420px，父元素的计算宽度是 328px。
图片匹配 `.event-card img { width: 420px; }`。

约束：
保留 HTML。不要裁切内容，也不要隐藏页面溢出。

请求：
最多列出三个原因并排序。为每个原因说明一项可以确认它的浏览器观察结果。
然后只针对已确认原因提出最小补丁。
```

这段提示限制了模型角色，并给出了可证伪的症状。

## 请求差异，而不是整块替换

处理现有代码时，要求模型：

- 只引用需要修改的行；
- 保留类名和公开结构；
- 用一句话解释每项改动；
- 指出所有假设；
- 提供回归检查清单。

之后分别应用并测试每个逻辑改动。小型差异仍可能有错，但更容易检查。

## 防范自信的编造

模型可能虚构浏览器支持、标准规则、无障碍要求、文件内容或测试结果。应根据不同主张采用不同方法：

| 主张 | 验证来源 |
| --- | --- |
| “这个选择器不匹配” | 实际 DOM 与 DevTools |
| “这个值胜出” | 计算样式追踪 |
| “这个特性得到广泛支持” | 当前兼容性来源和项目政策 |
| “这个表单无障碍” | 语义检查加键盘/人工测试 |
| “构建通过” | 真实的本地构建 |
| “用户偏好这个布局” | 研究或用户测试，而不是模型自信度 |

## 使用决策日志

```text
目标：
模型建议：
收集的证据：
决定：
拒绝的替代方案：
回归检查：
```

日志可以只有五行。它的作用是让作者身份和推理过程清晰可见。

## 保护私密与专有数据

遵循课程与组织政策。没有明确授权和获批服务时，不要把学生隐私信息、凭证、未公开代码或受限文档发送给模型。必要时对示例做脱敏处理，或使用本地/无 AI 工作流。

## 防止范围膨胀的审查提示

```text
只根据以下五项标准审查所提供的 HTML 和 CSS：
1. 原生元素选择
2. 键盘可见状态
3. 320px 至 1440px 范围内的溢出
4. 选择器优先级与耦合
5. 减少动态效果时的行为

返回一个表格，包含：
- 来自代码的证据，
- 风险，
- 最小建议测试，
- 置信度。

不要重写组件。不要虚构 JavaScript 行为。
```

## 知道何时不该问 AI

如果答案已经存在于 DOM、计算样式、Network 面板或需求中，直接检查即可。当前准确语法或支持情况很重要时，查阅可信资料。涉及产品意图、政策、评分或风险接受时，应由人类决策者决定。

## 结课项目的 AI 使用披露

在结课项目中，请说明：

- 哪些任务使用了 AI；
- 一项你接受并验证的建议；
- 一项你拒绝的建议；
- 一个通过浏览器证据诊断的缺陷；
- 如果没有模型，你会如何完成同样的验证。
