# 性能是可用性的一部分

<p class="lesson-lead">页面加载缓慢、阅读时发生位移或要求大量下载，都会把设备和网络受限的用户拒之门外。先处理最大的真实成本，并验证改进效果。</p>

## 优先优化用户所需内容

提出以下问题：

- 有意义的内容何时出现？
- 其他资源加载时，用户能否阅读和操作页面？
- 内容是否意外移动？
- 主图尺寸是否合适，优先级是否正确？
- 字体是否延迟文字显示或造成明显替换？
- 首个有用视图出现前需要多少 CSS？

性能指标可以提供帮助，但应将它们连接到具体体验与复现条件。

## 图像往往是最大的优化机会

1. 选择接近实际渲染需求的尺寸。
2. 使用合适格式和质量进行压缩。
3. 提供 `srcset` 和 `sizes` 以支持响应式选择。
4. 提供宽度和高度以预留宽高比。
5. 对首屏以下媒体延迟加载，但默认不要延迟加载主要内容图片。
6. 在 Network 面板中验证实际选择的请求和传输大小。

用 CSS 缩小一张 4,000 像素图片，并不会减少下载的字节。

## 保护布局稳定性

为图片、视频、广告和动态插入区域预留空间。除非已有专门预留位置，否则不要在读者当前位置上方插入横幅。使用稳定的字体回退并测试加载状态。

```css
.media-frame {
  aspect-ratio: 16 / 9;
  background: #e7e1d7;
}

.media-frame > img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

只有可以接受裁切时才使用 `object-fit: cover`。内容关键型图表可能需要 `contain` 或自然宽高比。

## 字体同时是产品与性能选择

系统字体不需要下载自定义字体。如果自定义字体很重要：

- 谨慎创建子集，同时保留所需语言和符号；
- 只选择实际使用的字重；
- 只预加载真正关键的文件；
- 有意图地选择 `font-display` 策略；
- 尽可能提供度量兼容的回退字体；
- 在慢速加载时检查文字渲染。

如果没有规划好的回退，不要为了极小的仅拉丁字母子集，牺牲中文、阿拉伯文或其他文字系统的可读性。

## 先保持 CSS 易懂，再考虑极致压缩

只有证据表明框架代码未使用时才将其移除。避免大量重复选择器和深层嵌套的生成规则。构建工具可以压缩生产 CSS，源码则应保持可维护。

关键 CSS 策略可以改善大型产品的首次渲染，却会增加复杂度和缓存失效风险。对于小型课程网站，直接减少 CSS 往往收益更大。

## 优先采用渐进增强

先用 HTML 提供内容和原生表单行为，再添加 CSS 呈现。只有交互确实需要时才添加脚本。网络失败不应抹掉本可存在于文档中的信息。

这种分层也有助于调试：每项增强都有清楚的边界和回退。

## 在可重复条件下测量

记录：

- 测试的页面或组件；
- 设备和浏览器类别；
- 视口；
- 模拟时的网络与 CPU 条件；
- 缓存状态；
- 指标或观察结果；
- 所做改动；
- 改动前后结果。

在开发者机器上快速加载一次，并不是有代表性的证据。

## 稳健性用例

测试：

- 图片禁用或加载缓慢；
- 样式表延迟；
- 自定义字体不可用；
- 长内容与翻译内容；
- 适用时离线返回已缓存的静态页面；
- 内容型页面的打印或阅读模式；
- 产品支持时的节省流量模式或受限网络。
