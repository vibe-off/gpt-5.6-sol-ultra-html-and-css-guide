# 根据证据调试，而不是凭冲动修改

<p class="lesson-lead">一套有效的调试方法，应在每一步之后缩小搜索范围。目标不是尝试许多修复，而是找出哪个解释能够经受浏览器验证。</p>

<div class="lesson-meta">
  <span>质量核心</span>
  <span>90 分钟</span>
  <span>浏览器证据</span>
</div>

## 六步循环

1. **复现：**用准确步骤、视口、内容和状态复现问题。
2. **缩减：**将案例缩小到仍能失败的最少 HTML、CSS 和条件。
3. **检查：**查看 DOM、匹配规则、计算样式、盒模型和相关无障碍状态。
4. **假设：**提出一个原因，并预测可观察的结果。
5. **修补：**只改变能够检验该假设的最小变量。
6. **回归：**重新检查原始需求，以及相邻状态、宽度、输入和内容。

如果补丁让症状消失，却无法解释原因，那么诊断还没有完成。

## 写出复现说明

较弱：

> 卡片在移动端坏了。

实用：

> 当视口宽度在 340px 到 386px 之间时，页面产生 92px 水平溢出。DevTools 显示图片在 328px 宽的卡片中计算为 420px。移除图片的固定宽度后，溢出消失。

第二种说法明确了条件、测量结果、元素和一项有用实验。

## 沿渲染管线检查

### 内容或结构问题

检查源码、解析后的 DOM、有效性、缺失的结束标签、意外嵌套、重复 ID 和 hidden 属性。

### 样式表或选择器问题

检查 Network 面板、文件路径、语法、选择器匹配、适用时的 Shadow DOM 或作用域边界，以及条件是否成立。

### 层叠问题

追踪生效和被划掉的声明、来源、重要性、层、选择器优先级、继承与源码顺序。

### 尺寸或布局问题

检查父子元素的计算尺寸、显示模式、可用空间、固有最小值、盒模型、包含块、溢出和布局覆盖层。

### 视觉层问题

检查层叠上下文、裁切、透明度、变换，以及伪元素是否覆盖目标。

### 交互或无障碍问题

检查原生元素选择、可聚焦性、顺序、无障碍名称、语义状态、隐藏行为和事件假设。

## 常见诊疗案例

| 症状 | 高价值证据 |
| --- | --- |
| 规则“没有作用” | 它匹配吗？值有效吗？哪条声明胜出？ |
| Flex 子元素溢出 | 自动最小尺寸、长内容、基准值、收缩 |
| Sticky 没有粘住 | 滚动容器、inset、可用滚动范围、祖先 overflow |
| z-index 总是失败 | 层叠上下文树，而不是更大的数字 |
| Grid 宽度不符合预期 | 轨道最小值、固有内容、gap、`minmax(0, 1fr)` |
| 表单标签无法激活输入框 | 相互匹配的 `for` 与唯一的 `id` |
| 焦点消失 | 轮廓线重置、裁切、离屏或隐藏状态 |
| 移动页面横向滚动 | 最宽元素、固定/固有宽度、变换、`100vw` |

## 缩减时不要破坏原因

复制出问题的组件，分小批移除无关兄弟元素、属性和状态。每次移除后确认错误仍然存在，最终保留仍能展示规则被破坏的最小示例。

缩减对人类与模型都有帮助。一个 20 行案例比带有无关组件的 2,000 行页面更能为调试伙伴提供有效信号。

## 将 DevTools 临时修改当作实验

禁用一条声明、修改一个值、强制一个状态、给盒子临时添加轮廓线、开关 Grid 或 Flex 覆盖层。这些改动是实验，不是最终源码。确认原因后，应在样式表中实现有明确意图的补丁，并在重新加载后复测。

## 让 AI 对假设排序

提供：

```text
复现步骤：
预期结果：
实际结果：
相关 DOM：
匹配/计算后的 CSS：
父元素约束：
已经进行的实验：
```

然后请求三个按可能性排序的假设、每个假设所预测的证据，以及区分它们的最小实验。这样能让模型保持诊断模式。

<div class="misconception">
  <h3>重写可能抹掉学习机会</h3>
  <p>替换整个组件也许能修复截图，却可能引入新的语义、优先级或边界问题。找出被破坏的关系后，应优先采用最小补丁。</p>
</div>

## 回归测试矩阵

修复后重新测试：

- 原始失败状态；
- 一个更窄和一个更宽的宽度；
- 键盘焦点与悬停；
- 短内容与长内容；
- 存在时的浅色/深色或偏好状态；
- 使用共享样式的相邻组件；
- 全新加载，而不只是 DevTools 临时修改。
