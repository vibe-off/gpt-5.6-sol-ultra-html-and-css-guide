# 质量不是收尾阶段

<p class="lesson-lead">当用户能够使用页面、需求在变化后仍能满足、缺陷可以诊断，并且其他开发者能理解其中的决策时，页面才算完成。</p>

## 审视每个页面的六个视角

<div class="concept-grid">
  <div class="concept-card">
    <h3>正确性</h3>
    <p>内容、链接、表单数据、状态和布局满足明确需求。</p>
  </div>
  <div class="concept-card">
    <h3>无障碍</h3>
    <p>语义、键盘操作、焦点、名称、标签、对比度、缩放和动态效果能够支持广泛用户。</p>
  </div>
  <div class="concept-card">
    <h3>稳健性</h3>
    <p>长内容、狭窄空间、媒体缺失、加载缓慢和偏好变化不会破坏体验。</p>
  </div>
  <div class="concept-card">
    <h3>可维护性</h3>
    <p>选择器、令牌、组件边界和注释让未来改动保持局部且容易理解。</p>
  </div>
  <div class="concept-card">
    <h3>性能</h3>
    <p>页面避免不必要的字节、渲染工作、布局偏移和阻塞资源。</p>
  </div>
  <div class="concept-card">
    <h3>证据</h3>
    <p>重要判断由检查、测试、标准或需求核对提供支持。</p>
  </div>
</div>

## 实现之前定义质量

将模糊目标转成可观察的标准：

| 模糊说法 | 可测试标准 |
| --- | --- |
| “适配移动端” | 320px 宽度下页面不出现水平滚动，控件仍易读且可操作 |
| “无障碍表单” | 标签可见、键盘顺序合理、焦点清楚、错误有关联、缩放后可用 |
| “快速图片” | 加载正确的候选源，尺寸预留空间，不传输不必要的过大文件 |
| “整洁的 CSS” | 没有无法解释的重要规则，共享决策使用令牌，改动限制在组件内部 |
| “AI 检查过了” | 已记录建议、浏览器证据、决定和理由 |

验收标准同时指导人类与 AI 的协作。没有这些标准，生成的视觉润色可能掩盖缺失的行为。

## 在整个构建过程中测试

进行小而频繁的检查：

- 完成 HTML 结构后：检查标题大纲、标签、链接和验证结果；
- 完成基础 CSS 后：追踪层叠、测试缩放、焦点和对比度；
- 完成布局后：测试窄宽视口、长内容和固有尺寸；
- 完成动态效果后：测试减少动态效果模式和键盘状态；
- 完成媒体后：检查传输大小、替代文本用途和布局稳定性；
- 重构后：确认外观和行为不变，同时规则系统更加简单。

## 缺陷也是课程内容

不要让完成后的方案掩盖每个错误。记录最小复现、揭示缺陷的证据和最小修复。由真实错误组成的案例库，比一组完美截图更有价值。
