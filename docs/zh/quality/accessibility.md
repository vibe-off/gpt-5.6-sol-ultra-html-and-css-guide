# 每一课都应包含无障碍

<p class="lesson-lead">无障碍衡量的是一种体验能否适应不同的身体、感官、输入方式、设备、环境和临时状况。它始于产品选择和语义化 HTML。</p>

## 从四项基础原则开始

### 1. 可感知

内容与状态需要人们能够感知的替代方式：有意义的替代文本、字幕、足够的对比度、可缩放文字，以及不只依赖颜色的提示。

### 2. 可操作

所有功能都必须能够在不要求精确指针操作的情况下使用。键盘焦点必须可见，顺序必须合理，目标必须易用，动态效果或时间限制不能形成障碍。

### 3. 可理解

导航、标签、说明和错误信息应清楚且一致。意外的上下文变化需要充分理由和明确沟通。

### 4. 稳健

使用有效的语义结构以及准确的名称、角色、状态和关系，让浏览器与辅助技术可以解释界面。

这些原则与 W3C《Web 内容无障碍指南》一致，但本页是一份实用的学习清单，并非合规审计。

## 优先使用语义化 HTML

原生元素自带角色和行为：

- 标题创建可导航的结构；
- `nav`、`main` 等地标元素创建区域；
- 链接与按钮表达不同结果；
- 标签与字段集建立表单关系；
- 表头标识数据关系；
- 对于合适内容，`details` 和 `summary` 提供原生展开模式。

当原生 HTML 无法表达所需界面时，ARIA 才有用，而且必须始终与行为同步。

## 不走捷径地进行键盘测试

把鼠标放到一边，完成真实任务。

检查：

1. `Tab` 和 `Shift+Tab` 能否按合理顺序到达每个交互控件？
2. 焦点是否始终可见？
3. 原生控件能否用预期按键激活？
4. 能否进入覆盖层、理解内容并离开，而不会困住焦点？
5. 发生重大变化后，焦点是否有意图地移动？
6. 隐藏内容是否同时退出视觉和焦点交互？

不要添加正数 `tabindex` 来强制匹配视觉顺序。应尽可能让 DOM、阅读、视觉与焦点顺序保持一致。

## 名称、角色、状态与值

检查交互元素的无障碍信息：

- **名称：**它叫什么？
- **角色：**它是哪种控件或区域？
- **状态：**展开、选中、禁用、勾选还是无效？
- **值：**范围控件或输入框当前包含什么？

一个只标为“更多”的按钮可能拥有按钮角色，但名称含糊。应修改可见文字，或提供符合上下文的名称，同时避免朗读标签与可见标签不一致。

## 将焦点指示设计成令牌

```css
:root {
  --focus-ring: #177f78;
}

:focus-visible {
  outline: 3px solid var(--focus-ring);
  outline-offset: 3px;
}
```

请在画布、卡片、图像、深色模式和错误表面上测试焦点环，避免 overflow 将它裁掉。焦点指示不是应该移除的浏览器缺陷。

## 缩放、文字与重排

浏览器缩放到 200% 并增大文字后：

- 文字不应被裁切或重叠；
- 控件应随标签增长；
- 除大型数据表等必要情况外，内容应重排，且不需要双向滚动；
- 粘性或固定区域不应占满整个视口；
- 阅读顺序与焦点顺序应保持连贯。

固定高度、绝对定位文字和视口大小的面板经常在这里失效。

## 对比度与非颜色提示

检查每种状态下文字与实际背景的对比度。弱化文字仍需易读。错误字段应同时使用颜色、文字和可见的结构提示；选中导航则可以结合颜色、字重、下划线、形状或 `aria-current`。

自动化对比度工具可以测量颜色对，但渐变、图像、透明度和组件状态仍需人工检查。

## 动态效果与感官特征

尊重 `prefers-reduced-motion`。不要只通过颜色、位置、形状或声音指导用户：“选择右侧绿色按钮”会在布局改变或颜色无法区分时失效。应使用控件的可见名称。

## 12 分钟课堂审查

1. **2 分钟：**浏览标题和地标。
2. **3 分钟：**用键盘完成主要任务。
3. **2 分钟：**检查三个无障碍名称和状态。
4. **2 分钟：**缩放到 200% 并检查重排。
5. **1 分钟：**启用减少动态效果。
6. **2 分钟：**检查一个表单错误和一个依赖颜色的状态。

这不能证明完全合规，但能培养习惯，并尽早发现影响大的问题。
