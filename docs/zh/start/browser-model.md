## 简化的处理流水线

可以把页面首次加载想象成一条由多个阶段相连的流水线：

1. 浏览器读取 HTML，并构建**文档对象模型（DOM）**。
2. 浏览器读取 CSS，收集可能应用于各元素的规则。
3. **层叠**决定哪些声明值胜出。
4. 继承和默认值补全其他值，浏览器由此产生**计算样式**。
5. 布局阶段计算各个盒子的尺寸与位置。
6. 绘制阶段把盒子变成像素，合成阶段再组合各个视觉图层。
7. 与此同时，语义信息会形成供辅助技术使用的**无障碍树**。

这是教学模型，并非完整的浏览器引擎规范，但它足以帮助你定位大多数初学者常见问题。

```html
<article class="notice">
  <h2>工作室已迁至 204 室</h2>
  <p>星期五请带上你的速写本。</p>
</article>
```

```css
.notice {
  max-inline-size: 36rem;
  padding: 1.5rem;
  border: 2px solid tomato;
}
```

HTML 提供了一个包含标题和段落的 `article`。CSS 提供了表现规则。由于该元素匹配 `.notice`，浏览器会把两者关联起来。

## 源代码、DOM 和像素是不同的证据

**源代码**是文件中的内容。**DOM** 是浏览器解析出的树；当标记无效或脚本对其做出修改时，它可能与源代码不同。**渲染后的页面**则是应用 CSS 与布局后的视觉结果。截图能够告诉你某处有问题，却通常不能指出问题产生在哪个阶段。

## 实用的诊断地图

| 症状 | 首先检查的位置 |
| --- | --- |
| 内容缺失或嵌套异常 | DOM 与 HTML 有效性 |
| 某条规则被划掉 | 层叠、优先级或无效值 |
| 值正确但看起来仍不对 | 计算样式、父元素约束或布局 |
| 内容被裁切 | 尺寸约束与溢出 |
| 键盘焦点无法到达控件 | 元素选择、焦点顺序或隐藏状态 |
| 屏幕阅读器读出的名称不明确 | 语义元素、文本、标签或无障碍名称 |

## 无障碍树不是可有可无的输出

有意义的 HTML 能在不添加任何 ARIA 的情况下提供角色与关系。`<button>` 天生具有按钮行为、键盘激活方式和按钮角色。一个可点击的 `<div>` 并不会因为 CSS 把它做得像按钮，就自动获得这些能力。

由此得到一条贯穿课程的规则：

> 使用 HTML 表达含义和行为，使用 CSS 控制表现。只有原生 HTML 无法表达所需语义时，才添加 ARIA。

## 引导式检查

打开任意文章页面，分别检查一个标题、一个链接和一张图片。

为每个元素记录：

1. 它在 DOM 中的父元素与子元素；
2. 一条匹配的 CSS 规则；
3. 一个计算值；
4. 它的盒子尺寸；以及
5. 如果适用，它的角色与无障碍名称。
