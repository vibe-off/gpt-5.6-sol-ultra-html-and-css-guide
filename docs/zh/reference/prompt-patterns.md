# 前端工作的 AI 提示模式

<p class="lesson-lead">复制结构、替换证据，并缩小请求范围。绝不要把模型表现出的自信直接写进证据日志。</p>

## 基础上下文块

```text
上下文：
期望行为：
实际行为：
相关 HTML/CSS：
浏览器证据：
约束：
我已经尝试过：
希望获得的帮助：
```

确实不适用的字段可以删除，但不要只是为了缩短提示而省略证据。

## 概念导师

```text
请使用以下内容解释[概念]：
1. 一个准确的心智模型；
2. 一个 15 行的 HTML/CSS 示例；
3. 一个要求先预测再运行的问题；
4. 一个常见误解；
5. 一项可以在 DevTools 中验证解释的观察。

假设我理解[已知概念]，但不了解[新概念]。
不要引入框架。
```

请验证示例和声称可观察到的结果。

## 语义清单

```text
请在编写标记前评审这些内容。

返回一个包含以下列的表格：
- 内容或操作；
- 用途；
- 推荐的原生 HTML 元素；
- 理由；
- 不确定之处或缺失的需求。

区分链接与按钮，以及语义章节与仅用于样式的包裹元素。
如果原生 HTML 已经能表达需求，不要添加 ARIA。
```

请用产品的实际行为核对这份清单。

## 按可能性排序的调试假设

```text
请作为调试伙伴，而不是代码重写工具。

[基础上下文和证据]

最多返回三个按可能性排序的假设。
每个假设应包含：
- 当前证据为什么支持它；
- 哪项浏览器观察可以确认它；
- 哪项观察可以否定它；
- 要运行的最小实验。

在观察结果确认原因之前，不要提出补丁。
```

## 层叠追踪

```text
请解释这一个元素上的这一个属性最终胜出的声明。

元素和 DOM 上下文：
按源码顺序排列的匹配声明：
层叠层和重要性：
计算结果：

依次检查相关性、来源/重要性、层叠层、特异性和源码顺序。
一旦某个阶段已经决定胜者，就停止继续比较。
```

请将回答与 DevTools 中的计算值追踪进行比较。

## 最小补丁

```text
以下证据已经确认了原因：
[证据]

请提出满足下列条件的最小补丁：
- 保留 HTML 和公开类名；
- 不裁剪内容；
- 不添加 !important；
- 不为文本引入固定高度；
- 满足这些验收标准：[标准]。

仅返回：
1. 一个小型 diff；
2. 每项改动的一句理由；
3. 三项回归检查。
```

约束必须符合真实任务；请删除不相关的约束。

## 无障碍风险评审

```text
请从以下方面评审提供的 HTML/CSS 是否存在潜在问题：
- 原生元素选择；
- 无障碍名称和状态；
- 键盘顺序和可见焦点；
- 标签和表单错误；
- 对比度和非颜色提示；
- 缩放/重排；
- 减少动态效果。

每一项都引用代码中的证据，并标记为：
- 已由代码确认；
- 需要浏览器检查；
- 需要产品需求澄清。

不要只凭源码评审就声称组件已经无障碍。
```

## 对抗性内容案例

```text
根据这些布局约束和内容字段，提出五个可能暴露脆弱性的边缘情况。

至少包括：
- 很长的本地化标题；
- 无法断行的字符串；
- 缺失或加载缓慢的媒体；
- 狭窄容器；
- 200% 缩放。

不要重写组件。说明每种情况下预期的稳健行为。
```

## CSS 可维护性评审

```text
只分析：
- 选择器耦合；
- 特异性热点；
- 重复的设计决策；
- 不清晰的组件职责；
- 只解释机制、不解释原因的注释。

先返回证据和风险。在我选择一个已验证的热点前，不要建议重构。
```

## 决策日志

```text
目标：
模型建议：
收集的证据：
决策：
被拒绝的方案：
回归检查：
```

## 提示词反模式

| 避免 | 改为 |
| --- | --- |
| “让它更好” | 验收标准和一个评审视角 |
| “修复所有 CSS” | 一个可复现症状和按可能性排序的原因 |
| “它无障碍吗？” | 明确的人工检查和源码评审项目 |
| “使用最佳实践” | 项目的浏览器、内容和维护约束 |
| “给我生产级代码” | 小型 diff、测试和声明的假设 |
| 粘贴私有项目 | 脱敏后的最小案例或已批准的本地工作流 |

## 最终原则

提示词不是证据，回答也不是证据。你的检查、测试、标准和需求验证才是证据。
