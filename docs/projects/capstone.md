# Capstone: Campus hub

<p class="lesson-lead">Build a small site for a real or fictional campus organization. The result should be useful, resilient, inspectable, and easy to change—not merely screenshot-ready.</p>

<div class="lesson-meta">
  <span>Capstone</span>
  <span>8–12 hours</span>
  <span>Build + defend</span>
</div>

## Choose one organization

Examples:

- campus film society;
- student research showcase;
- volunteer tutoring group;
- design exhibition program;
- outdoor activities club; or
- department workshop series.

Use realistic content and requirements. Do not use trademarked branding in a way that implies an official site.

## Required pages

### Home

- clear organization identity and purpose;
- primary navigation;
- featured event or call to action;
- short introduction to three activities; and
- contact or participation route.

### Events or resources

- at least six structured items;
- meaningful headings, metadata, and links;
- responsive collection layout; and
- one data table or other structured information when genuinely appropriate.

### Join or register

- clear instructions;
- name, contact, and relevant choice controls;
- visible labels and help text;
- native input types and validation baseline;
- understandable error and success design; and
- privacy-conscious wording appropriate to a classroom prototype.

No real personal data needs to be collected or transmitted. The project may remain a static prototype.

## Technical requirements

### HTML

- complete document shell on every page;
- unique titles and one clear `h1` per page;
- consistent landmarks and primary navigation;
- current-page indication;
- meaningful link text and valid relative paths;
- purposeful media alternatives and dimensions; and
- native elements before custom semantics.

### CSS

- consistent tokens for core visual decisions;
- low-specificity component styling;
- normal flow as the base;
- purposeful Flexbox and Grid;
- fluid content plus content-led breakpoints;
- visible focus and complete interaction states;
- reduced-motion behavior; and
- no unexplained `!important` or page-level overflow clipping.

### Quality

- no unexplained browser console or resource errors;
- keyboard completion of navigation and form;
- readable at 200% zoom;
- usable from about 320px through a wide desktop;
- long-content test;
- one current compatibility decision for a modern feature; and
- a concise evidence and AI disclosure log.

## Process milestones

1. **Brief and content inventory** — what must exist and for whom?
2. **Semantic wireframe** — page regions, heading outline, links, controls, data.
3. **Unstyled HTML review** — content order, semantics, paths, form relationships.
4. **Visual system** — tokens, type, spacing, color, focus.
5. **Layout pass** — normal flow, component layouts, responsive constraints.
6. **Quality pass** — keyboard, zoom, long content, motion, media, performance.
7. **Live change** — teacher-selected requirement.
8. **Defense** — demo, browser evidence, tradeoffs, AI disclosure.

## Required evidence

- heading outline for each page;
- one computed-style trace for a cascade decision;
- one box or layout measurement for a responsive fix;
- keyboard task notes;
- zoom and reduced-motion notes;
- one compatibility source and fallback decision;
- one before/after performance or transfer observation;
- one AI suggestion accepted and verified;
- one AI suggestion rejected with reason; and
- one defect reduced to a minimal case.

## Live change pool

The instructor chooses one after the first complete version:

- Add a seventh navigation destination without horizontal overflow.
- Double the featured-event title and summary length.
- Place the event card inside a 300-pixel sidebar container.
- Add a second form choice with conditional help text.
- Create a dark theme using existing role tokens.
- Replace the hero image with a portrait source and preserve useful cropping.
- Change the accent color while maintaining contrast and state distinction.

The student has 25–40 minutes to plan, implement, test, and explain the change.

<div class="ai-brief">
  <h3>AI policy</h3>
  <p>AI may support planning, generation, critique, and debugging. The student must disclose material use, verify all claims, and complete the live change and defense. Fabricated evidence or tests are treated as missing work.</p>
</div>

<ul class="acceptance-list">
  <li>All pages communicate clearly with CSS disabled.</li>
  <li>The complete primary journey works by keyboard.</li>
  <li>Responsive behavior follows content constraints and preserves logical order.</li>
  <li>CSS responsibilities and precedence remain understandable.</li>
  <li>Quality evidence is concrete and reproducible.</li>
  <li>The student can modify and explain the final system.</li>
</ul>

## Assessment

Use the detailed [rubrics and assessment guide](/projects/rubrics). A polished visual result cannot compensate for broken semantics or an unusable primary task; equally, semantic correctness should still be presented with care.
